# Customer Data Onboarding Copilot

## v1.3 Product Requirements And Technical Specification

- Status: Implemented and locally verified
- Release name: Contract-Driven Onboarding And Reconciliation
- Decision date: 2026-07-11
- Depends on: MVP v1, v1.1 audit readiness, and v1.2 exception explainability
- Primary stack: Streamlit, pandas, PostgreSQL, OpenAI API, ReportLab

## 1. Release Summary

V1.3 turns the current single-schema onboarding demo into a repeatable, contract-driven workflow. Users will be able to select or import an approved target data contract, configure deterministic transformation pipelines, forecast and reconcile PostgreSQL publishing, and correct rejected rows through an auditable reprocessing loop.

The release contains four connected capabilities:

1. Configurable, versioned target data contracts.
2. A controlled transformation rule builder.
3. Pre-publish and post-publish reconciliation.
4. Rejected-row correction and reprocessing.

Together, these capabilities answer four real-world questions:

- What exact target contract was used?
- How was each source value converted?
- Did every source and target record get accounted for?
- How were rejected records corrected and safely reprocessed?

## 2. Current Baseline And Product Gap

The existing application already supports:

- A fixed healthcare eligibility target schema with three canonical tables.
- Source profiling and type alignment.
- Rules-based and AI-assisted mapping suggestions.
- Human mapping approval.
- Schema-versioned mapping template save/load.
- Blocking validation errors and warnings.
- Canonical transformation and PostgreSQL upserts.
- Import replay detection and reviewer signoff.
- Rejected-row exports with original values.
- Field-level lineage.
- HTML and PDF reports.

The remaining gap is operational repeatability. The fixed target schema is defined in application code, transformations are not configurable as ordered rule pipelines, reconciliation does not distinguish forecasted inserts/updates/unchanged records from actual writes, and rejected records do not yet have a controlled correction loop.

## 3. Goals

V1.3 must:

- Allow analysts to select a published target contract or import a JSON contract.
- Persist contracts and immutable published versions in PostgreSQL.
- Tie every mapping template and import run to an exact contract version.
- Let reviewers construct deterministic transformation pipelines from an approved operation catalog.
- Preview transformed sample values before mapping approval.
- Retain step-level transformation evidence in field lineage.
- Calculate pre-publish insert, update, unchanged, duplicate, and integrity expectations.
- Compare the forecast with actual PostgreSQL results.
- Roll back publishing when hard reconciliation checks fail.
- Support inline and CSV-based corrections for rejected source records.
- Reprocess only selected rejects when source values change.
- Reprocess the complete source when mapping or transformation logic changes.
- Preserve immutable original values, correction history, lineage, and reviewer signoff.

## 4. Non-Goals

V1.3 will not include:

- A visual target schema designer.
- Automatic schema creation from database introspection.
- Mutable published contract versions.
- Arbitrary Python, SQL, JavaScript, or user-authored expressions.
- External API or database lookup transformations.
- Automatic execution of AI-suggested transformations.
- Hard validation error waivers.
- Multi-file onboarding packages.
- Authentication, role-based access, or multi-tenant isolation.
- Object-storage retention of every uploaded source file.
- Background jobs, scheduling, or recurring feed monitoring.
- Large-file processing beyond the current Streamlit and pandas design.
- Schema drift or value-distribution drift monitoring.

## 5. Primary Users

### 5.1 Schema Owner

- Imports a target contract.
- Reviews contract validation results.
- Publishes or retires a contract version.
- Defines contract-level reconciliation policies.

V1.3 captures the schema owner's name as audit metadata but does not implement authentication or permissions.

### 5.2 Onboarding Analyst

- Selects a published contract.
- Profiles a customer file.
- Reviews mappings and transformation rules.
- Runs validation and resolves rejected records.
- Reviews reconciliation before publish.

### 5.3 Reviewer Or Approver

- Reviews warnings, mappings, transformation evidence, and reconciliation.
- Records signoff and comments.
- Approves publishing for original or recovery runs.

## 6. Locked Product Decisions

| Area | V1.3 decision |
| --- | --- |
| Contract input | Built-in contracts plus JSON import/export. |
| Contract persistence | PostgreSQL contract registry. |
| Contract lifecycle | Draft, published, and retired. |
| Published versions | Immutable. Changes require a new version. |
| Mapping compatibility | Exact contract version match. |
| Transformation execution | Controlled deterministic catalog only. |
| Multi-source transformations | Coalesce and concatenate are supported. |
| Split transformations | One source may supply multiple target fields through explicit split rules. |

| Reconciliation warnings | Permit publish only after reviewer signoff. |

| Mapping or rule correction | Version the template and rerun the complete source file. |
| Hard validation errors | Cannot be waived. |
| Unresolved rejects | May be acknowledged with a comment but are never published. |

## 7. End-To-End Workflow

1. Select a published target contract or import a new draft JSON contract.
2. Validate and, when authorized by the demo workflow, publish the contract version.
3. Upload a source CSV or load the synthetic demo file.
4. Profile source columns.
5. Generate rules-based or AI-assisted mapping suggestions.
6. Configure and preview deterministic transformation pipelines.
7. Approve mappings and transformations as one mapping-template version.
8. Validate all rows against the selected contract.
9. Correct source-level errors or acknowledge unresolved rejects.
10. Transform accepted rows and generate lineage.
11. Run pre-publish reconciliation against PostgreSQL.
12. Capture reviewer signoff when the run has warnings.
13. Publish in one transaction.
14. Run post-publish reconciliation and persist actual outcomes.
15. Generate HTML, PDF, JSON, CSV, and lineage artifacts.
16. For later corrections, create a child run linked to the original run.

## 8. Epic 1: Configurable Target Data Contracts

### 8.1 Contract Definition

A target data contract is a versioned JSON document that describes the expected canonical output. It must include:

- Stable contract key.
- Display name and domain.
- Semantic version.
- Tables.
- Fields and descriptions.
- Data types.
- Required and nullable settings.
- Primary keys and business keys.
- Foreign-key relationships.
- Allowed values.
- Generated-field strategies.
- Validation rules and severities.
- Reconciliation policy.

Example:

```json
{
  "contract_key": "healthcare_eligibility",
  "name": "Healthcare Eligibility Canonical",
  "domain": "healthcare",
  "version": "1.1.0",
  "tables": [
    {
      "name": "members",
      "primary_key": ["member_id"],
      "business_key": ["member_id"],
      "foreign_keys": [],
      "fields": [
        {
          "name": "member_id",
          "data_type": "identifier",
          "required": true,
          "nullable": false,
          "generated": false,
          "allowed_values": [],
          "validation_rules": [
            {
              "kind": "member_identifier",
              "severity": "error",
              "parameters": {}
            }
          ],
          "description": "Stable unique identifier for the member."
        }
      ]
    }
  ],
  "reconciliation_policy": {
    "max_reject_rate": 0.10,
    "max_reject_rate_severity": "warning",
    "require_zero_orphans": true,
    "require_exact_row_accounting": true
  }
}
```

### 8.2 Supported Contract Vocabulary

V1.3 must support the current canonical data types:

- text
- identifier
- enum
- date
- numeric
- email
- phone
- boolean

The contract may reference only validation kinds implemented in the application's validation registry. Import must fail with an actionable message when a contract uses an unsupported data type, validation kind, generation strategy, or reconciliation setting.

### 8.3 Contract Lifecycle

Contract versions use these states:

| State | Meaning |
| --- | --- |
| draft | May be replaced or deleted before use in a published import. |
| published | Selectable for onboarding and immutable. |
| retired | Retained for history but unavailable for new onboarding runs. |

Lifecycle requirements:

- Contract key plus version must be unique.
- Publishing requires a structurally valid contract.
- A published contract cannot return to draft.
- Retiring a contract does not alter historical runs or templates.
- Changes to a published contract require importing or creating a new version.
- New import runs may select only published versions.
- Historical runs must continue to render the contract version they used.

### 8.4 Contract Import And Export

The Target step must provide:

- A dropdown of published contracts.
- Contract name, version, domain, and status.
- JSON upload for a new draft contract.
- Validation results before persistence.
- JSON download for the selected contract.
- Publish and retire commands with actor name and comment.
- Existing table dropdown and field inspection for the selected version.

Import must reject:

- Malformed JSON.
- Missing required contract metadata.
- Duplicate contract key and version.
- Duplicate table or field names.
- Missing business keys for publishable target tables.
- Foreign keys that reference unknown tables or fields.
- Generated fields without a supported strategy.
- Unsupported types, validations, or reconciliation policies.

### 8.5 Contract Version Binding

- Mapping templates must store `contract_key`, `contract_version`, and contract checksum.
- Loading a template against another version must fail safely and explain the mismatch.
- Import runs and reports must store the same identifiers.
- Contract checksum mismatch for a published version is a hard integrity error.
- The current healthcare schema must be seeded as the first built-in published contract.

### 8.6 Epic 1 Acceptance Criteria

- A valid JSON contract can be imported as draft and stored in PostgreSQL.
- An invalid contract produces field-specific validation errors and is not stored.
- A draft can be published and selected for onboarding.
- A published contract cannot be edited.
- A new version can be created without changing the old version.
- Retired versions remain visible in import history but cannot start a new run.
- Mapping templates load only for the exact contract version and checksum.
- The existing healthcare demo works through the seeded contract without behavioral regression.

## 9. Epic 2: Transformation Rule Builder

### 9.1 Transformation Model

Each approved target mapping may contain an ordered transformation pipeline.

```json
{
  "target_table": "plans",
  "target_field": "plan_type",
  "source_columns": ["Plan Type"],
  "steps": [
    {"operation": "trim", "parameters": {}},
    {"operation": "uppercase", "parameters": {}},
    {
      "operation": "map_values",
      "parameters": {
        "mapping": {
          "P.P.O.": "PPO",
          "HIGH DEDUCTIBLE": "HDHP"
        }
      }
    }
  ],
  "failure_policy": "warning_set_null"
}
```

Pipelines execute in their displayed order. The output from one step is the input to the next step.

### 9.2 Approved Operation Catalog

| Operation | Purpose | Required parameters |
| --- | --- | --- |
| trim | Remove leading and trailing whitespace. | None. |
| null_if | Convert configured tokens such as blank, N/A, or NULL to null. | Values. |
| uppercase | Convert text to uppercase. | None. |
| lowercase | Convert text to lowercase. | None. |
| title_case | Convert text to title case when explicitly selected. | None. |
| regex_replace | Replace a controlled text pattern. | Pattern and replacement. |
| parse_date | Parse accepted date formats and emit ISO date values. | Accepted formats. |
| parse_numeric | Parse numeric or currency-like values. | Decimal and grouping settings. |
| normalize_phone | Remove formatting and normalize a phone value. | Default country setting, optional. |
| normalize_email | Trim and normalize email casing. | None. |
| map_values | Convert source values through an explicit crosswalk. | Value mapping. |
| default | Use a configured value only when the current value is null. | Default value. |
| coalesce | Select the first non-null value from ordered source columns. | Ordered source columns. |
| concatenate | Join multiple source values. | Source columns, separator, null behavior. |
| split | Select a configured part from a delimited source value. | Delimiter and part index. |
| generate_deterministic_id | Generate an ID from declared business-key inputs. | Input fields and strategy. |

`generate_deterministic_id` is available only when the selected contract marks the target field as generated.

### 9.3 Failure Policies

| Policy | Behavior |
| --- | --- |
| error | Add a blocking validation issue and reject the row. |
| warning_set_null | Add a warning and produce null. Allowed only for nullable targets. |
| warning_keep_original | Add a warning and retain the previous value when type compatible. |
| use_default | Use an explicitly configured default and record the substitution. |

Default behavior:

- A failed conversion into a required target field is an error.
- A failed conversion into an optional nullable field is a warning and null.
- Unknown enum values follow the contract validation rule's severity.
- Defaults are never inferred silently.

### 9.4 Mapping And Transformation UI

The Map step must let a reviewer:

- Select one target field at a time.
- Inspect target type, nullability, allowed values, and validation rules.
- Select one or more source columns when the chosen operation permits it.
- Add, remove, and reorder approved transformation steps.
- Configure step parameters through structured controls.
- Preview original, intermediate, and final values for representative rows.
- See transformation failures before approval.
- Save the mapping and transformation pipeline as one template version.

The UI must not accept executable code or free-form formulas.

### 9.5 Multi-Source And Split Rules

- `coalesce` and `concatenate` may consume multiple source columns.
- Each multi-source rule must preserve the ordered source-column list in lineage.
- One source column may map to multiple targets through separate mapping rules.
- A split mapping must explicitly select its delimiter and target part.
- A target field receives one approved pipeline. Competing pipelines block approval.

### 9.6 AI-Assisted Transformation Suggestions

AI mode may return operation IDs and structured parameters from the approved catalog. The application must:

- Reject unknown operation IDs.
- Validate parameters before display.
- Mark every AI-generated step as suggested.
- Require reviewer approval before execution.
- Never execute model-generated code.
- Record AI rationale separately from deterministic execution evidence.

### 9.7 Transformation Lineage

Field lineage must retain one row per source record and target field while adding:

- source_columns
- source_values_json
- corrected_values_json, when applicable
- transformation_trace_json
- final_value
- mapping_template_version
- contract_version

`transformation_trace_json` stores an ordered list containing operation ID, input value, output value, status, and message for every step. Reports may display a summarized trace while CSV and database records retain the complete trace.

### 9.8 Epic 2 Acceptance Criteria

- A reviewer can build, reorder, preview, approve, save, and reload a pipeline.
- Repeated execution of the same input and approved pipeline produces the same output.
- Multi-source coalesce and concatenate preserve source order.
- Split rules can map one source value into separately approved target fields.
- Invalid step parameters block approval.
- Arbitrary code cannot be entered or executed.
- AI suggestions outside the approved catalog are rejected.
- Required and optional conversion failures follow the documented policies.
- Lineage shows every applied operation and before/after value.
- Changing an approved pipeline creates a new mapping-template version.

## 10. Epic 3: Publish Reconciliation

### 10.1 Reconciliation Stages

V1.3 supports two stages:

1. Pre-publish forecast against the current PostgreSQL target state.
2. Post-publish comparison of forecasted and actual outcomes.

### 10.2 Run-Level Accounting

The following invariant is always required:

```text
source rows = accepted rows + rejected rows
```

The run summary must include:

- Source file hash.
- Source rows.
- Accepted rows.
- Rejected rows.
- Acknowledged rejected rows.
- Rows with warnings.
- Contract key and version.
- Mapping-template name and version.
- Parent run, when applicable.
- Mapping mode.
- Reviewer and signoff state.

Acknowledged rejects remain part of the rejected count.

### 10.3 Target-Table Forecast

For each target table, calculate:

- Candidate rows from accepted source records.
- Unique business keys.
- Exact duplicate candidates removed safely.
- Conflicting duplicate candidates.
- Expected inserts.
- Expected updates.
- Expected unchanged records.
- Missing required outputs.
- Foreign-key orphans.

An existing business key counts as unchanged only when all publishable target values match the current database record. Key existence alone is insufficient.

Because one source row may contribute to several target tables and dimension tables are deduplicated, target-table counts are not expected to equal accepted source-row counts. Reconciliation must use contract business keys, relationships, and field lineage.

### 10.4 Reconciliation Statuses

| Status | Meaning | Publish behavior |
| --- | --- | --- |
| PASS | All hard checks pass and no warning policy is exceeded. | Publish may proceed after normal signoff requirements. |
| WARNING | Hard checks pass but a contract tolerance is exceeded. | Publish requires explicit reviewer signoff. |
| FAIL | At least one hard integrity check fails. | Publish is blocked or rolled back. |

Hard failures include:

- Source row accounting mismatch.
- Missing required target output for an accepted source record.
- Conflicting duplicate business keys.
- Foreign-key orphan where the contract requires integrity.
- Contract or mapping-template version/checksum mismatch.
- Database write failure.
- Forecasted versus actual insert, update, or unchanged mismatch.
- Post-publish integrity check failure.

Warnings include:

- Reject rate above the contract threshold.
- Optional contract tolerance exceeded.
- Acknowledged unresolved rejects.
- Existing non-blocking validation warnings.

The default `max_reject_rate` is `0.10` with warning severity. Contracts may override the value and severity within the supported reconciliation policy.

### 10.5 Transaction Behavior

- Pre-publish reconciliation must complete before database writes begin.
- Canonical table writes must occur in one PostgreSQL transaction.
- Post-write checks must run before commit where technically possible.
- A hard post-write failure rolls back all canonical writes for that publish attempt.
- The failed reconciliation result must still be recorded in audit storage after rollback.
- Audit evidence must distinguish blocked-before-write from rolled-back-after-write.
- A successful transaction records actual inserts, updates, and unchanged records.

### 10.6 PostgreSQL-Unavailable Behavior

When PostgreSQL is unavailable:

- Source and transformed-output reconciliation remain available.
- Database insert/update/unchanged forecasts are marked unavailable.
- Publishing remains disabled.
- Reports clearly label the result as transform-only reconciliation.
- Transform-only reconciliation cannot claim `Published` status.

### 10.7 Reconciliation UI And Exports

The Publish step must show:

- Overall forecast status.
- Run accounting equation.
- One selectable target-table summary at a time.
- Expected insert, update, unchanged, duplicate, and orphan counts.
- Blocking failures and warnings.
- Contract tolerance values.
- Reviewer signoff requirement.

After publish, the same view must show expected versus actual values and transaction status.

The Report step must export reconciliation as:

- HTML within the validation report.
- PDF within the validation report.
- JSON containing complete machine-readable checks and metrics.

### 10.8 Epic 3 Acceptance Criteria

- Pre-publish reconciliation correctly classifies insert, update, and unchanged records.
- Exact duplicate candidates are counted separately from conflicting duplicates.
- Sparse duplicates coalesce missing optional values, while distinct non-null values for the same field remain conflicts.
- Source accounting mismatches block publish.
- Orphan relationships block publish when required by the contract.
- A warning threshold requires signoff but does not force failure.
- A hard write or post-write mismatch rolls back canonical writes.
- Successful actual counts match forecast counts.
- Failed reconciliation evidence remains auditable after rollback.
- Transform-only reconciliation works without PostgreSQL and cannot publish.
- HTML, PDF, and JSON outputs use the same reconciliation data model.

## 11. Epic 4: Rejected-Row Correction And Reprocessing

### 11.1 Immutable Source Identity

At ingestion, every source row receives:

- `source_record_id`: SHA-256 of source file hash plus original row number.
- `source_row_number`: original one-based file row number including the header offset already used by validation.
- `original_row_fingerprint`: SHA-256 of a canonical serialization of the original row.

The original values and fingerprint are never modified by correction workflows.

### 11.2 Correction Modes

V1.3 supports:

1. Inline source-value correction in the validation workspace.
2. Export and upload of a correction CSV.
3. Mapping or transformation correction through a new template version.

Users may edit source values and mapping/transformation configuration. They may not edit system metadata, generated IDs, or canonical table records directly.

### 11.3 Correction CSV

The correction export must contain:

- source_record_id
- source_row_number
- original_row_fingerprint
- error_codes
- error_source_columns
- error_target_fields
- original source columns prefixed with `original__`
- editable correction columns prefixed with `corrected__`
- correction_comment

Corrected columns are prefilled with the original values. On upload, the application must:

- Require all system identity columns.
- Reject unknown or duplicate source_record_id values.
- Verify the original row fingerprint.
- Ignore attempted changes to `original__` values.
- Apply only changed `corrected__` values as overlays.
- Require a correction comment for every changed record.
- Show upload validation errors before saving corrections.

### 11.4 Source-Value Correction Flow

For inline or CSV source-value corrections:

1. Select one or more rejected records.
2. Edit source values or upload correction overlays.
3. Save correction reason and reviewer identity.
4. Revalidate only the selected rejected records.
5. Create a child run with `run_kind=row_correction`.
6. Keep newly valid records as recovered.
7. Keep unresolved records rejected.
8. Run reconciliation for recovered records against the current database state.
9. Capture new reviewer signoff.
10. Publish only recovered records through idempotent upserts.

Previously accepted rows are not reprocessed for source-value-only corrections.

### 11.5 Mapping Or Transformation Correction Flow

Changing a mapping, enum crosswalk, parsing rule, or any other transformation can affect previously accepted rows. Therefore:

- The change creates a new mapping-template version.
- The complete original source file must be reprofiled, remapped as needed, transformed, validated, and reconciled.
- The new child run uses `run_kind=full_rerun`.
- The source file hash must match the parent run.
- If the source dataframe is no longer in the active session, the user must reupload the original file.
- The reuploaded file must pass hash verification before processing.
- Idempotent upserts classify resulting records as inserted, updated, or unchanged.

V1.3 does not persist full raw accepted source files for later reruns. It stores hashes and rejected raw payloads. Production object storage is a future enhancement.

### 11.6 Correction Statuses

| Status | Meaning |
| --- | --- |
| rejected | Record has at least one unresolved hard validation error. |
| correction_draft | A correction overlay exists but has not been revalidated. |
| pending_revalidation | Correction is queued in the current child run. |
| recovered | Corrected record passes validation but is not yet published. |
| still_rejected | Corrected record still has a hard error. |
| acknowledged_reject | Reviewer accepts that the record will remain rejected. |
| published_recovery | Recovered record was published successfully. |

Acknowledging a reject requires a reviewer name and comment. It does not waive errors and never makes the record publishable.

### 11.7 Signoff And Audit

- Every child run links to one parent import run.
- Every correction stores source record, source column, original value, corrected value, reason, reviewer, and timestamp.
- Hard validation errors cannot be waived.
- Warnings may proceed through signoff.
- Recovered rows require fresh signoff before publish.
- Reports must distinguish original, corrected, recovered, still rejected, and acknowledged records.
- Field lineage must show original value, corrected value, transformation trace, and final target value.

### 11.8 Epic 4 Acceptance Criteria

- A reviewer can correct one rejected row inline without changing its original values.
- A valid correction CSV can update selected rejected rows.
- A correction CSV with an unknown ID, duplicate ID, or fingerprint mismatch is rejected.
- Only selected rejected rows are revalidated for source-value corrections.
- Previously accepted rows are not republished during a recovery-only run.
- Mapping or transformation changes force a full-file child run.
- A later full rerun requires the original file hash to match.
- Hard errors cannot be waived.
- An acknowledged reject remains rejected and unpublished.
- Recovered records require fresh signoff and reconcile before publish.
- Parent-child run history and correction lineage are visible in reports.

## 14. Audit And Versioning Rules

- Historical contracts, templates, runs, reconciliation results, and corrections are append-only.
- Published contract definitions are never updated in place.
- Approved mapping-template changes create a new template version.
- Child runs never overwrite parent run results.
- Reports render stored versioned definitions rather than the latest contract.
- Reviewer names and comments remain explicit metadata until authentication is introduced.
- Timestamps use UTC in persistence and may be localized for display.

## 15. Non-Functional Requirements

### 15.1 Determinism

- Approved transformations produce identical results for identical inputs and configuration.
- Reconciliation calculations are deterministic for a fixed database snapshot.
- Generated IDs remain stable for the same declared business key.

### 15.2 Safety

- No transformation executes arbitrary code.
- Hard reconciliation failures cannot be bypassed in the UI.
- Canonical writes are transactional.
- Original correction evidence is immutable.

### 15.3 Explainability

- Every mapping, transformation, validation result, correction, and publish outcome has human-readable evidence.
- Reports can identify the exact contract and template versions used.

### 15.4 Privacy

- V1.3 continues to use synthetic demo data.
- Correction values are not newly sent to the LLM.
- AI transformation suggestions use the existing sanitized mapping context and approved operation vocabulary.
- Full raw accepted source files are not persisted by this release.

### 15.5 Performance

- The 1,000-row synthetic demo must remain responsive in the Streamlit workflow.
- Reconciliation may query PostgreSQL in batches rather than once per record.
- Transformation tracing must not change output correctness when trace display is limited.
- V1.3 makes no production-scale throughput claim.

### 15.6 Backward Compatibility

- Existing healthcare demo behavior remains available through the seeded built-in contract.
- Existing v1.1/v1.2 reports and import history remain readable.
- Existing mapping templates must either migrate to the seeded contract version or fail with an actionable compatibility message.
- Database changes are additive and must not delete existing canonical or audit records.

## 17. Testing Strategy

### 17.1 Contract Tests

- Import a valid contract.
- Reject malformed and structurally invalid contracts.
- Reject unsupported data types and validation rules.
- Enforce lifecycle transitions and immutability.
- Enforce exact template-to-contract version matching.
- Preserve historical rendering after a later version is published.

### 17.2 Transformation Tests

- Test every approved operation independently.
- Test ordered operation composition.
- Test null and failure policies.
- Test coalesce, concatenate, and split behavior.
- Test deterministic generated IDs.
- Reject arbitrary or unknown operations.
- Validate AI suggestions against the operation catalog.
- Verify step-level lineage values.

### 17.3 Reconciliation Tests

- Verify source row accounting.
- Classify inserts, updates, and unchanged records against seeded PostgreSQL data.
- Distinguish exact and conflicting duplicates.
- Coalesce a missing optional value with a known value without masking distinct non-null conflicts.
- Detect missing outputs and foreign-key orphans.
- Trigger warning thresholds and required signoff.
- Trigger hard failures and verify rollback.
- Verify failed reconciliation audit evidence survives rollback.
- Verify forecast and actual agreement.
- Verify transform-only reconciliation without PostgreSQL.

### 17.4 Correction And Reprocessing Tests

- Apply an inline correction and preserve the original value.
- Round-trip a correction CSV.
- Reject identity and fingerprint mismatches.
- Revalidate only selected rejected rows for value corrections.
- Keep unresolved rows rejected.
- Mark acknowledged rejects without publishing them.
- Require full rerun for mapping or transformation changes.
- Verify original file hash before a later full rerun.
- Verify child-run linkage and fresh signoff.
- Verify recovered publishing remains idempotent.

### 17.5 Regression Tests

- Run the existing 1,000-row demo end to end.
- Keep all existing profiler, mapper, validation, transform, report, and idempotency tests passing.
- Verify HTML and PDF generation remains non-empty and readable.
- Verify current rejected-row and field-lineage exports remain available.

## 19. Release Success Measures

V1.3 is successful when:

- A new valid contract can be imported and used without changing Python schema code.
- Every run references immutable contract and mapping-template versions.
- Reviewers can explain the complete transformation chain for any target field.
- Pre-publish forecasts match actual PostgreSQL outcomes in successful runs.
- Injected accounting or integrity mismatches reliably block or roll back publish.
- A rejected source record can be corrected, revalidated, signed off, and published without republishing previously accepted rows.
- A transformation change reliably forces a complete, versioned rerun.
- All existing v1.2 functionality and tests remain intact.

## 20. Risks And Mitigations

- Configurable contracts could expand into a general-purpose schema platform. Limit v1.3 to JSON contracts and an approved vocabulary; defer visual authoring and introspection.
- Transformation rules could become an unsafe scripting system. Use a closed operation registry with structured parameters and no executable expressions.
- Insert/update/unchanged forecasts could drift from concurrent database state. Run publish reconciliation in one transaction and treat forecast/actual mismatch as a hard failure.
- Correction workflows could silently create mixed processing logic. Reprocess only row-value corrections selectively; force full reruns for template changes.
- Persisting correction data increases privacy exposure. Store only required rejected payloads and correction audit data; do not persist complete accepted source files in v1.3.
- Step-level lineage can produce wide or verbose exports. Store the complete ordered trace as JSON and show summarized evidence in the UI and PDF.

## 21. Release Acceptance Criteria

The release is complete only when all of the following are true:

- Contract registry, lifecycle, JSON import/export, and exact version binding work.
- The healthcare schema runs from the registry rather than a hard-coded-only path.
- Transformation pipelines use the approved catalog and support preview, approval, persistence, and lineage.
- No arbitrary code path exists in transformation configuration.
- Pre-publish and post-publish reconciliation produce consistent shared report data.
- Hard reconciliation failures prevent partial canonical writes.
- Warning reconciliation requires reviewer signoff.
- Inline and CSV source-value correction paths work.
- Selected reject recovery does not republish prior accepted rows.
- Mapping or transformation changes trigger full source reruns.
- Parent-child runs, corrections, reconciliation, and signoff are auditable.
- HTML, PDF, JSON, CSV, and lineage artifacts include the required v1.3 evidence.
- Database migration works against existing v1.2 data.
- Black, Ruff, pytest, compilation, and the complete local demo verification pass.
- README and architecture documentation describe the new workflow and its production limitations.
